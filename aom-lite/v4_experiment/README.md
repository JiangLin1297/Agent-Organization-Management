# V4 实验：组织管理学 vs 无组织管理

## 实验目的

**核心问题**：同样的任务，用组织管理学来管AI Agent，到底有没有比不用管理学更好？

V3实验的局限性在于：它只是按能力高低给Agent贴了"新手/专家"标签，然后给不同标签的人不同风格的指令。这本质上是在测试"因材施教有没有用"，不是在测试"组织管理学有没有用"。

组织管理学的核心不是"对不同的人说不同的话"。组织管理学的核心是：
- **职能分工**：谁负责什么角色？（产品经理、架构师、前端开发、后端开发、测试）
- **汇报关系**：谁向谁汇报？谁协调谁？（科层制）
- **协作流程**：任务怎么分解？怎么在角色之间传递？谁做最终审核和整合？

V4实验直接测试这三个维度。

## 实验设计

### 对照组（无组织管理）

```
用户 ──→ 单个Agent ──→ 游戏代码
         （全栈开发）
```

- **只有一个Agent**，没有角色分工、没有协作、没有组织结构
- 模拟普通人现在使用AI的方式：一个聊天窗口，丢任务进去，等结果
- Agent同时承担产品经理、架构师、前端开发、后端开发、测试的全部职责
- 一次API调用，直接输出最终结果

### 实验组（有组织管理）

```
用户 ──→ 协调者 ──→ 产品经理 ──→ 架构师 ──→ 前端开发 ──┐
                    （需求定义）  （技术设计）  （界面实现）  │
                                              ──→ 游戏逻辑开发 ──→ 测试 ──→ 协调者
                                                  （核心逻辑实现）  （质量审查）  （最终整合）
```

- **6个Agent**，按组织职能分工
- **明确的汇报关系**：所有角色向协调者汇报
- **结构化的协作流程**（7步）：
  1. 协调者分解任务
  2. 产品经理定义需求
  3. 架构师设计技术方案
  4. 前端开发实现界面（并行）
  5. 游戏逻辑开发实现核心逻辑（并行）
  6. 测试审查代码
  7. 协调者整合最终产出

## 组织结构设计

### 角色职责

| 角色 | 职责 | 向谁汇报 | 管理原则 |
|------|------|----------|----------|
| **协调者** | 任务分解、进度管理、最终整合 | 用户 | 例外管理：只关注偏离标准的产出 |
| **产品经理** | 功能需求、UX标准、验收准则 | 协调者 | A3报告：结构化需求文档 |
| **架构师** | 代码结构、模块划分、技术选型 | 协调者 | 满意化：选择"足够好"的方案 |
| **前端开发** | HTML结构、CSS样式、界面交互 | 协调者 | 偏差报告：只实现非标准部分 |
| **游戏逻辑开发** | 蛇的移动、碰撞检测、计分系统 | 协调者 | 偏差报告：只标注关键算法 |
| **测试** | Bug检测、功能验证、代码审查 | 协调者 | 例外管理：只报告问题 |

### 协作流程

```
步骤1  协调者分解任务 ─────────────────────────────────────→
步骤2  产品经理定义需求 ←─────────────────────────────────
步骤3  架构师设计技术方案 ←───────────────────────────────
步骤4  前端开发实现界面 ────────┐（并行）
步骤5  游戏逻辑开发实现逻辑 ───┤（并行）
步骤6  测试审查代码 ←───────────┘
步骤7  协调者整合最终产出 ←───────────────────────────────
```

### §2.5管理原则在各角色中的应用

| 管理原则 | 应用角色 | 具体体现 |
|----------|----------|----------|
| A3报告（结构约束） | 产品经理、架构师 | 输出严格按固定板块和字数限制组织 |
| 例外管理（偏差过滤） | 测试、协调者 | 只报告偏离标准的问题，正常的不报告 |
| 满意化（停止规则） | 架构师、前端开发 | 选择"足够好"的方案，不过度优化 |
| 信息过滤（价值层次） | 所有角色 | 高价值信息详细写，低价值信息省略 |

## 运行方式

### 前置条件

1. 安装Python依赖：
```bash
pip install openai
```

2. 设置环境变量：
```bash
export OPENAI_API_KEY=your_api_key
export OPENAI_BASE_URL=https://token-plan-cn.xiaomimimo.com/v1  # 可选
export LLM_MODEL=mimo-v2.5-pro  # 可选
```

### 运行对照组

```bash
cd aom-lite/v4_experiment/control_group
python run.py
```

输出：
- `output/game.html` — 生成的贪吃蛇游戏
- `output/experiment_log.json` — 实验日志（时间、Token等）

### 运行实验组

```bash
cd aom-lite/v4_experiment/experimental_group
python run.py
```

输出：
- `output/game.html` — 整合后的贪吃蛇游戏
- `output/artifacts/` — 各阶段中间产出（需求文档、架构设计、代码片段等）
- `output/experiment_log.json` — 实验日志（每个Agent的时间、Token等）

### 在浏览器中查看结果

```
# 对照组
start aom-lite/v4_experiment/control_group/output/game.html

# 实验组
start aom-lite/v4_experiment/experimental_group/output/game.html
```

## 理论基础

本实验直接测试AOM框架的核心主张：

1. **韦伯的科层制**（§2.1）：实验组采用层级分工结构，对照组无层级
2. **法约尔的管理职能**（§2.1）：实验组体现了计划（PM）、组织（架构师）、协调（协调者）、控制（测试）
3. **明茨伯格的构型理论**（§2.1）：实验组的角色分工映射为信息角色（PM/架构师）和人际角色（协调者）
4. **§2.5的管理沟通约束**：A3报告、例外管理、满意化、信息过滤在各角色prompt中具体实现
