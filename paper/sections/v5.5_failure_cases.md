# 7.9 Failure Cases and Limitations

While OIMAC achieves substantial efficiency gains, we observed several recurring failure modes across experimental runs. Documenting these failures is essential for a realistic assessment of the approach and for identifying directions for improvement.

## Failure Case 1: Color Mapping Inaccuracy (Cross-Role Interface Mismatch)

**Problem.** In several runs (particularly Conditions C and D), the perler bead pattern generator produced grids where the displayed colors did not match the original image. Users reported that the color statistics table showed correct counts, but the rendered grid used a different color palette.

**Root cause.** The image processing agent and the frontend agent independently implemented color quantization. The image processing agent used K-means clustering in RGB space, while the frontend agent applied a different quantization method (median cut) for display. Since each agent only received the other's output through the pipeline — without a shared color mapping specification — the two implementations diverged.

**Mechanism.** This is an instance of the **Incremental Role Context (IRC) limitation**: while IRC reduces context size by passing only relevant information, it can inadvertently omit interface-critical details. The color palette specification was considered "implementation detail" and was not included in the interface contract between the two agents.

**Observed frequency.** This issue appeared in approximately 20% of Condition C runs and 30% of Condition D runs across both models.

**Mitigation.** In Condition C (with Context Controller), the coordinator's final integration step includes a cross-agent consistency check that catches and resolves some (but not all) of these mismatches. A more robust solution would require explicit interface specifications — a structured contract between roles that defines shared data formats.

## Failure Case 2: Canvas Rendering Failure (Agent Output Quality Variance)

**Problem.** In MiMo Condition B (iterative refinement), run 4 produced an output where the Canvas rendering mechanism was non-functional. The automated quality evaluation scored this run 20/25, with `has_drawImage=false` and `has_pixel_rendering=false` — the grid was rendered using a fallback table-based method rather than Canvas.

**Root cause.** During iterative refinement, the agent's fourth-round output introduced a regression: a code refactoring that replaced the Canvas-based rendering with a simpler table-based approach. The agent did not detect this regression because it lacked a mechanism to verify that each refinement preserved core functionality.

**Mechanism.** This illustrates a fundamental limitation of iterative self-improvement: **each round optimizes for the agent's internal evaluation criteria, which may not align with the user's requirements**. Without explicit quality gates, iterative refinement can introduce regressions in pursuit of other improvements (e.g., code simplicity).

**Observed frequency.** Canvas rendering failures appeared in 2 of 40 automated quality evaluations (5%), both in iterative conditions (B).

**Mitigation.** OIMAC's pipeline structure with a dedicated Tester role partially addresses this by introducing an explicit review step. However, the Tester's evaluation is also based on the agent's internal criteria, not a formal specification. Future work should explore automated regression testing within the pipeline.

## Failure Case 3: Coordination Overhead in Fine-Grained Decomposition

**Problem.** When we experimented with decomposing the task into more than 7 roles (e.g., separating "color quantization" and "grid rendering" into distinct agents), total token consumption increased rather than decreased. The additional coordination cost outweighed the benefits of finer specialization.

**Root cause.** Each additional pipeline stage introduces fixed overhead: the new agent must receive and process its input context, produce output, and that output must be passed to downstream agents. When the task is already well-decomposed, further subdivision adds more coordination cost than it saves in per-agent context reduction.

**Mechanism.** This relates to the **Coordination Overhead Minimization (COS)** mechanism: there exists an optimal team size k* where the marginal benefit of decomposition equals the marginal cost of coordination. Beyond k*, adding more roles is counterproductive.

**Observed frequency.** This was observed in exploratory runs (not part of the main experiment) where we tested 10-role and 12-role configurations.

**Implication.** OIMAC's efficiency gains are not monotonic in the number of roles. The choice of k=7 for our experiments was informed by task complexity, but the optimal k* is task-dependent and currently determined by heuristics rather than formal analysis. Future work should develop methods for automatically determining the optimal decomposition depth.

---

## Summary

These failure cases illustrate three important caveats about OIMAC:

1. **Interface specification is critical.** Pipeline decomposition reduces context but can lose cross-role interface details. Explicit interface contracts between roles are needed.

2. **Quality assurance in pipelines is non-trivial.** While OIMAC's Tester role catches some issues, automated regression testing within the pipeline remains an open problem.

3. **Decomposition has diminishing returns.** There exists an optimal team size; over-decomposition increases coordination cost without reducing per-agent context.

OIMAC introduces new coordination challenges alongside efficiency gains. These challenges do not negate the efficiency benefits — all failure cases occurred within contexts where OIMAC still outperformed baselines — but they highlight areas for future improvement.
