# 1. Introduction

## 1.1 The Problem: Context Accumulation in Iterative Multi-Agent Systems

The rapid adoption of large language model (LLM) agents has created an engineering paradox. Individual LLM agents are remarkably capable — they can write code, analyze data, and generate creative artifacts. Yet when organizations attempt to scale from single agents to multi-agent systems, costs escalate in ways that defy intuition. A seven-round iterative refinement of a web application, for instance, consumes over 90,000 tokens — fifteen times more than a single-pass generation of comparable quality.

We identify the root cause as the **Context Accumulation Effect (CAE)**: in any architecture where an LLM agent iteratively refines its own output, each refinement round must embed the complete artifact from the previous round as input context. Since the artifact grows with each round, the total token consumption follows a superlinear — and in the worst case, quadratic — growth pattern. CAE is not a bug in any particular implementation; it is a structural property of iterative self-improvement architectures.

The economic implications are significant. At current API pricing, a single iterative workflow can cost 10–15x more than a single-pass generation. As organizations deploy multi-agent systems at scale — with dozens of agents collaborating on complex tasks — CAE imposes a hard economic ceiling on what is practically achievable.

## 1.2 Existing Approaches and Their Limitations

The dominant response to context growth has been to treat it as a compression problem. Recent work on prompt optimization (Zhou et al., 2022), context distillation (Snell et al., 2024), and retrieval-augmented generation (Lewis et al., 2020) all share a common assumption: the problem is that there is *too much context*, and the solution is to make context *smaller*.

These approaches are valuable but fundamentally incomplete. Compression reduces the *volume* of context without addressing its *topology*. In an iterative refinement chain, even a compressed representation of the full history must still be processed by each subsequent step. The computational structure — a linear chain where every node depends on every predecessor — remains unchanged. This means that while compression can reduce the constant factor, it cannot change the growth rate.

A second class of approaches introduces multiple agents but preserves the shared-context architecture. Frameworks such as AutoGen (Wu et al., 2023) and CrewAI enable multi-agent collaboration through shared conversation threads, where all agents observe all prior outputs. While this distributes the workload, it does not reduce the total context processed — in fact, it often increases it, because every agent must read the full history regardless of its specific role.

## 1.3 Core Insight: Restructuring Computation, Not Reducing Context

We propose a fundamentally different diagnosis: the problem is not the *amount* of context but the *structure* of computation. In an iterative chain, all computation flows through a single bottleneck — the agent must process the entire history to produce the next refinement. This is analogous to a manufacturing process where every worker must inspect every preceding worker's output before adding their own contribution. The solution in manufacturing was not to make inspection faster (compression) but to reorganize the production line so that each worker receives only what they need (specialization and handoff).

This insight — that **restructuring computation is more powerful than reducing context** — is the foundation of our approach. We show that by decomposing a monolithic iterative process into a pipeline of specialized roles, where each role receives only its direct dependencies, the quadratic growth of CAE can be reduced to linear growth.

## 1.4 Method: OIMAC as Structural Solution

We introduce **OIMAC** (Organizational Intelligence Multi-Agent Coordination), a coordination protocol that instantiates this insight as a concrete system design. OIMAC decomposes a task into a pipeline of specialized roles (e.g., requirements analysis, architecture design, implementation, testing), where:

1. Each role receives only the artifacts it directly depends on (not the full history)
2. A **Context Controller** enforces bounded context passing with configurable size limits
3. The pipeline structure ensures that no single role ever processes the cumulative output of all preceding roles

OIMAC is not a prompt engineering technique or a context compression heuristic. It is a **structural solution** that changes the computational topology of multi-agent coordination. The key mechanism — which we term **Computational Decomposition Effect (CDE)** — eliminates CAE by ensuring that context growth is bounded by the size of individual role outputs, not by the cumulative history.

## 1.5 Contributions

This paper makes four contributions:

1. **Problem formalization**: We identify and formalize the Context Accumulation Effect (CAE), demonstrating that iterative LLM self-improvement incurs Ω(n²) token cost growth — a structural pathology independent of model capability or prompt design.

2. **Mechanism decomposition**: Through controlled experiments, we decompose the efficiency gains of organizational coordination into two measurable components: Computational Decomposition Effect (CDE, ~70% of gains) and Context Controller (CC, ~30% of gains).

3. **Structural solution**: We present OIMAC as an executable coordination protocol that reduces iterative cost from Ω(n²) to O(k), validated across two heterogeneous models and two heterogeneous tasks with statistical significance (p < 0.001, Cohen's d > 5.6).

4. **Real-world baseline comparison**: We demonstrate that OIMAC outperforms not only idealized baselines but also realistic multi-agent architectures (simulating AutoGen/CrewAI-style systems), achieving ~1.4x further cost reduction through structured computation.
